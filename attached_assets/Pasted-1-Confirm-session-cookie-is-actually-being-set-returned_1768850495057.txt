1) Confirm session cookie is actually being set + returned in production

On GET /api/auth/login and on callback, inspect response headers in production:

Must see Set-Cookie: connect.sid=...; HttpOnly; Secure; SameSite=None

Cookie must be present on subsequent /api/auth/user calls.

If cookie is missing: fix cookie config / trust proxy.

Required server settings

app.set("trust proxy", 1) must be active in prod (it is in our code, keep it)

Session cookie config for prod:

secure: true

sameSite: "none"

httpOnly: true

add proxy: true in session options (helps behind proxies)

Also ensure all frontend fetch calls use credentials (looks like apiRequest() does, but confirm no direct fetch without it):

credentials: "include"

2) Make SESSION_SECRET stable across deployments (this causes endless “session expired”)

If SESSION_SECRET is missing in deployment secrets OR changes between deploys, every cookie becomes invalid.

Action

Set SESSION_SECRET in Replit Deployments “Secrets” to a long random value and never change it.

3) Fix domain strategy mismatch for Replit auth (REPLIT_DOMAINS must include prod hostnames)

Your auth setup registers strategies based on REPLIT_DOMAINS and chooses strategy by req.hostname.

Action
Set REPLIT_DOMAINS to include every production hostname you use, comma-separated, for example:

4sgraphics.replit.app

quote.4sgraphics.com (if you still use it)

any other custom domain you hit in production

If this is wrong, callback can end in /?error=no_user or repeated auth failures.

4) Kill “SESSION EXPIRED” retry storms (UI/Client fix)

Right now console shows repeated 401s (dashboard + now-mode/current). That means the UI keeps re-requesting after auth has failed.

Action

Add a global API interceptor: if any request returns 401, stop all polling/retries, clear cached queries, and show ONE “Session expired → Log in” screen.

Don’t keep calling /api/dashboard/* and /api/now-mode/* in background after 401.

5) Service Worker is likely caching stale JS in production (common Replit bug)

Your console logs show SW manager + “Cache Version”. If the service worker is serving an older bundle, it can break prod-only.

Action options (pick one)

Easiest: disable service worker in production entirely.

Or: bump cache version on every deploy AND force clients to refresh (skipWaiting + clientsClaim) AND on login/logout clear caches.

6) Reports/Charts blank only in production = Odoo config/env missing or failing silently

Dev works, prod doesn’t → 90% chance production environment is missing these (names may differ in your code):

ODOO_URL

ODOO_DB

ODOO_USERNAME

ODOO_PASSWORD (or API key)

any report-specific model endpoints

Action

Add /api/odoo/health (server-side) returning:

configured: boolean

canAuth: boolean

In Reports/Charts UI:

if configured=false, show “Odoo not configured in production” instead of blank screen.

if canAuth=false, show error and log exact server exception.
